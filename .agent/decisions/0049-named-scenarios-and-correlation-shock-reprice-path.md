# 0049 — Named historical scenarios compose as compound shocks; the correlation family reprices through a second path

Status: accepted (2026-06-15)
Area: analytics, risk & QC (extends ADR 0006's scenario grid)

## Context

TARGET §5.4 prescribes four stress axes — spot, vol, rate, **correlation** — plus a catalogue
of **named historical scenarios** (2008, COVID-March-2020). The spot/vol/rate axes and the
rate-axis precedent ([rate axis], ADR 0006) all reprice one way: shock the market *state* and
re-run the option pricer (`full_reprice_pnl`). The two new families do not fit that mould
cleanly, and a later agent would otherwise have to reverse-engineer two non-obvious choices.

## Decision

### 1. A named scenario is one *compound* `Scenario`, not a new mechanism

Each named scenario is a single `Scenario` with `family="named"`, id `named_<label>`, carrying a
**joint** spot/vol/rate shock applied together — a whole crisis day at once, not the parametric
grid's one-axis-at-a-time sweep. It reuses the existing `spot_shock`/`vol_shock`/`rate_shock`
fields and the existing `shock_valuation` → `price` reprice verbatim, so it lands in the same
`full_reprice_pnl` cell as every other family and needs no new repricing path. It is appended
**last** in the grid (after the parametric block) so the parametric cells stay byte-identical.
The catalogue lives in config (`NamedScenarioConfig`, `scenarios.yaml`); the shipped 2008/COVID
magnitudes are **owner-tunable seeds** sized to the broad shape of each crisis, not calibrated to
a named source.

### 2. The correlation family reprices through `basket_variance`, not the pricer — and is dormant

A ρ̄ bump does **not** move any single option's price; it moves the *basket's* implied vol via the
Eq-23 identity (`basket.py::basket_variance`). So the correlation family deliberately reprices
through a **second path**: `correlation_shock_pnl(BasketCorrelationExposure, scenario)` recomputes
the basket vol under the bumped ρ̄ (a genuine reprice, never a Taylor multiplier) and monetizes the
basket-vol change against the book's dollar basket-vega. The `Scenario.correlation_shock` field
defaults to `0.0`, so this path is inert for every spot/vol/rate/time/named cell — only a
`correlation`-family (or a named scenario with a correlation leg) cell drives it.

**The family is built but dormant.** The live option book carries no ρ̄ exposure (that needs
constituent capture + the signal layer, both downstream), so a ρ̄ bump reprices to zero through the
pricer. We spec-and-wire the axis now — parallel family, empty-by-default, exercised only against a
constructed `BasketCorrelationExposure` in tests — so the §5.4 fourth axis exists the day the
exposure becomes real. A ρ̄ bump that drives the basket non-PSD raises `NonPSDBasketError` (carried
up from `basket_variance`) rather than silently flooring.

### 3. Construction-hash discipline: additive-when-non-empty, config-bundle-moves-by-design

Both families follow the rate-axis pattern: `_grid_construction_hash` folds `correlation_shocks`
and `named_scenarios` into the payload **only when non-empty**, so the *grid* construction hash
(`effective_scenario_version`) is byte-identical for an unconfigured grid — the single invariant
the families are built around. The *config-bundle* hash (`config_hashes["scenarios"]`) does move
when the fields are added (a bundle hash sees the new keys, empty or not), with section isolation
preserved (universe/qc/pricing untouched); that golden was regenerated by design.

## Consequences

- Two reprice paths now coexist in the engine. The README is explicit that `full_reprice_pnl`
  serves spot/vol/rate/time/named (the pricer) and `correlation_shock_pnl` serves the correlation
  family (the basket-variance identity); they are not interchangeable.
- The correlation cell is a zero on the live book until ρ̄ exposure lands — by design, not a bug.
- BFF/front wiring of both families is deferred (front-adjacent, owner-ruled like the rate axis):
  `tasks/frontend-named-scenarios-wiring.md`.
