# Routing map — where things live

This is a routing table, not a description. It points; the detail lives in each
directory's `README.md`. It stays short and rarely goes stale because directory
locations change far less often than file contents. When you add or move a
top-level area, update this file in the same change.

| Area | Path | What it owns | Start here |
|------|------|--------------|------------|
| Monorepo | `packages/` + `apps/` | The layered uv-workspace — the whole system, one tree, one gate (ADR 0018). `packages/core` = `algotrading.core` (config/log/manifest/provenance — level 0). `packages/infra` = `algotrading.infra`: the **frozen contract seam** `infra/contracts` (table contracts, instrument key, `StorageRepository` protocol — M0 owns it) plus the full stack — the market-data plane (`connectivity`, `universe`, `collectors`), the pure-function analytics core (`pricing`, `snapshots`, `forwards`, `iv`, `surfaces`), the risk engine (`risk` — ADR 0006), QC + its sibling anomaly/triage plane (`qc` + `validation`, collapsed into one persisted `triage_records` table — ADR 0010), and the actor-driven operable layer (`orchestration` — jobs, qc_job, the five metrics, four alerts, dashboard, run-state ledger, EOD pipeline, the `reconstruction` subpackage — plus `observability` run-lineage over the `RunRegistry`). One driver, the actor, runs live and replay, which is what makes the headline acceptance tests (byte-identical replay, provenance, reconstruction, handover) pass inside the root gate. Nautilus is the runtime spine ([ADR 0023](decisions/0023-nautilus-runtime-spine-and-library-leverage.md)): IBKR rides Nautilus's adapter plus a custom Client-Portal REST transport (ADR 0024/0025); the collection seam is unified on the push `RawCollector` ([ADR 0027](decisions/0027-collection-seam-push-canonical.md)). `packages/infra-ibkr` = the broker leaf adapter — **IBKR is the sole live broker** (Saxo/Deribit removed; index-only, [ADR 0042](decisions/0042-index-options-only-scope-ibkr-sole-broker.md)); the provider dimension stays generic ([ADR 0017](decisions/0017-provider-dimension.md)) so another broker could rejoin. `packages/{strategy,execution}` + `apps/frontend` (Python BFF + React/Vite web) = the upper layers. Layering enforced by import-linter; the only gate is the root `uv run ruff/mypy/lint-imports/pytest`. Each module's `README.md` is the next hop (mirrored under `documentation/modules/`). | `.agent/decisions/0018-monorepo-keystone-m0.md` |
| Operations | `documentation/` | Operator handover (E): the five lifecycle runbooks (`documentation/runbooks/`), the **connectivity & deployment guides** (`documentation/connectivity/` — connect a provider, schedule capture, stand up unattended server collection; distinct from the runbooks, which operate an already-running system), release-management rules + release notes (`documentation/releases/`), the frozen interface-contract list, the config & reproducibility standard (`configuration-and-reproducibility.md`, ADR 0028), and known limitations / support model. `documentation/modules/` mirrors every per-directory `README.md` via symlink, so each module's doc lives next to its code *and* under one roof here. Start at the index. | `documentation/README.md` |
| Plan of record (next phase) | `documentation/roadmap-index-analytics.md` | **What we build next.** The merge/convergence is closed; the active target is the **index options-analytics pipeline** (index → point-in-time constituents → Δ-band chain → IV/surface/Greeks in decimal+dollar → daily-close cron → operator front page), sequenced Phase 0→3. Pairs with `vision-medium-term.md` (the why) and `.agent/open-questions.md` (decisions — OQ-1…6 now all ruled). The blueprint still overrides on any domain/formula conflict (ADR 0011). | `documentation/roadmap-index-analytics.md` |
| Scripts | `scripts/` | Operator CLI tooling above the workspace (surface plotting, sample export/reconstruct, figure/PDF export). **Not** a uv-workspace member and **not** in the root gate — run with `uv run` (plotting/export tools need `--group notebooks`). | `scripts/README.md` |
| Configs | `configs/` | Economic + operational config as the six blueprint Part VII YAML bundles (`environment`/`broker`/`universe`/`qc`/`scenarios`/`pricing`); `load_platform_config(configs/)` builds the typed `PlatformConfig` from the four economic bundles. Per ADR 0028 the standard is YAML profiles → typed validated config → DI into pure compute; config is as-of/effective-dated and frozen per run. No business parameter as a `.py` literal. | `documentation/configuration-and-reproducibility.md` |

Read `00-overview.md` first; Part II for math, Part III for the roadmap, Part IV for implementation guides. This is the plan of record (ADR 0011); `AGENTS.md` governs process, the blueprint governs domain — they do not conflict. | `documentation/blueprint/README.md` |
