## 1. Findings priorisés (post-réfutation)

Légende collision : **🟢 zone libre (backend)** = pas de conflit avec Matthieu · **🟠 web/ (lane Matthieu)** = coordonner.

### P0 — faux chiffres / fonction morte

**P0-1 · Positions · Greeks $ sans multiplicateur de contrat → sous-estimation ×10 (SX5E)** 🟢
`positions_read.py:135` fait `dollar[name] * quantity` en **oubliant `multiplier`** (la ligne 134 pour
`position` l'applique pourtant). Tous les Greeks $ affichés et la somme du book sont **10× trop petits**
pour SX5E. `test_positions_api.py:268` **encode le bug**.
*Fix :* `dollar[name] * quantity * multiplier` ; régénérer fixture + golden.

**P0-2 · Strategy · décision de sortie/FLATTEN évaluée puis jetée → kill-switch mort en simulation** 🟢
`engine.py:130` appelle `run_strategy()` et **jette le `StrategyStep`** ; `decide_exit()`
(`s2_put_line.py:118-136`) calcule `ExitAction.FLATTEN` au `exit_delta_ceiling` mais `_run_day` ne le lit
jamais. Le paramètre `exit_delta_ceiling` n'a **aucun effet** sur le backtest.
*Fix :* capturer `step` ; si `step.exit_.action == FLATTEN`, solder le book avant de ré-évaluer l'entrée.

**P0-3 · Operations · IBKR `status="unavailable"` codé en dur → run impossible après login réussi** 🟢/🟠
`providers.py:43` littéral figé jamais mis à jour ; `run.py:33` le renvoie tel quel ;
`RunControlPanel.tsx:37` grise si `status!='ready'` ; `runner.py:118-120` rejette aussi tout provider ≠
SAMPLE. Après login IBKR OK, l'option reste grisée et `POST /api/run?provider=IBKR` renvoie 409.
*Fix :* brancher `_status_payload()` (`ibkr.py`) dans `list_providers` → `ready` quand `established` ;
étendre `runner.py` à IBKR ; MAJ `Operations.test.tsx:154` (🟠 test en web/, à coordonner).

### P1 — trompeur ou faux à fort impact

**P1-1 · Market · toggle Strict/Indicatif cosmétique — le BFF ignore `mode`** 🟠 *(Matthieu live)*
`Market.tsx:97` envoie `&mode=…` (+`:98 &run_id=`) ; `analytics.py:448-452` `get_analytics()` n'a pas le
paramètre → FastAPI **les drop silencieusement**. Données **identiques** dans les 2 modes ; seuls la
légende et la fraction de coverage changent. Le PM croit ajouter des marks one-sided. *(Le redesign est
déjà spec : MAT-LEGIBILITY-strict-indicative-mode ; l'inertie du toggle est l'observation neuve.)*
*Fix court :* retirer le toggle (ou son label) tant que l'engine ne le supporte pas.

**P1-2 · Positions · vue datée = fills du jour marqués aux derniers prix → valeur chimère** 🟢
`ledger.read(trade_date=)` en égalité stricte (`ledger.py:50`) → une date historique ne montre que les
fills **de ce jour** (une jambe ouverte la veille disparaît). Et `_latest_pricing_by_key`
(`positions_read.py:104-110`) prend le snapshot **le plus récent toutes dates** → fills J marqués prix
J+N. « All booked dates » implique cumulatif : la vue mono-date est sémantiquement cassée.
*Fix :* retirer le filtre date (positions = cumulatives) **ou** vraie sémantique as-of
(`fills.trade_date<=date AND mark.date()<=date`).

**P1-3 · Strategy · Sharpe calculé sur P&L en devise brut, pas sur des rendements** 🟢
`results.py:100,106` passe une liste de `realized_pnl` absolus à `annualised_sharpe()` (valide seulement
pour une série de rendements). Avec multiplier 10, valeurs O(centaines) → **Sharpe en milliers**, rendu
verbatim (`BacktestResults.tsx:47`).
*Fix :* normaliser chaque P&L par le MTM/notional de début de jour ; ou masquer si `|Sharpe|>100`.

**P1-4 · Basket · le stressor classe la jambe ATM put comme call → mauvais facteur d'actualisation** 🟢
`basket_scenarios.py:69-79` `_option_right(target_delta)` renvoie 'C' si `target_delta>=0` ; `atm` **et**
`atmp` ont `target_delta=0.0` mais `atmp` est un PUT. `reconstruct_valuation` reprice un CALL et divise
le prix PUT stocké par lui → DF corrompu. Le template **Straddle** utilise une jambe `atmp` → **tout
stress de straddle est corrompu**.
*Fix :* utiliser `delta_band` (présent sur la row, `tables.py:234`) pour le right, logique déjà dans
`projection.py`.

**P1-5 · Operations · jour warn-QC peint en rouge « QC fail », et les 2 chemins QC se contredisent** 🟢/🟠
`pipeline.py:122` écrit `OUTCOME_FAILED` pour `overall_status!='pass'` (inclut `warn`) ;
`recorded_dates.py:18-19` `failed→fail→` badge rouge. **Mais** `health.py:22-24` teste contre
`QC_FAIL_STATUSES` (`store_reads.py:8`) qui **omet `warn`** → même jour = pastille verte. **Le PM voit
« QC Passing » vert en haut et « QC fail » rouge dans le tableau, pour la même date.** *(Racine = bug
ADR-0052 sévérité déjà confirmé ; le symptôme neuf = la contradiction inter-panneaux.)*
*Fix :* `OUTCOME_WARN` quand `warn` ; 3ᵉ état ambre traversant `recorded_dates`, `api.ts` QcVerdict, `health.py`.

**P1-6 · Signals · les lignes-compagnon d'iv_rank rendues comme panneaux de magnitude ; caption « 1-year » sur une fenêtre de 3 jours** 🟢/🟠
`serializers.py:418-427` : `iv_rank_n_observations`(=4) et `iv_rank_window_days`(=3) tombent dans le
défaut et passent en `by_kind` → barres « IV rank n observations = 4.0 » comme si c'étaient des signaux.
`api.ts:528` hardcode « its 1-year range » alors que la donnée vit sur 3 jours / 4 obs — caption fausse
**exactement quand le backfill est mince** (le cas pour lequel le mécanisme compagnon a été construit).
*Fix :* filtrer les kinds-compagnon au BFF en bloc metadata ; caption dynamique « {N}-day, {k} obs ».

**P1-7 · Market · ticker constituant → copie « marché probablement fermé » + contrôles morts** 🟠 *(Matthieu live)*
`Market.tsx:94-99` fetch `/api/analytics?underlying=<subject>` où subject peut être un constituant ;
seule la dispersion garde l'indice. Pour un constituant : `maturities:[]`, `surfaceMissing`, et
`describeSurface` émet « No two-sided quote for ASML… market probably closed » → **faux** : la surface
est absente **par design ADR 0051**. *(La suppression de TickerSelector par Matthieu résout sans doute
une partie — re-vérifier après son commit.)*
*Fix :* fetcher analytics/signals avec `index` (pas `subject`) ; ou garder SurfacePanel/TenorPanel sur `kind==='index'`.
