> **LANDED & ARCHIVED (2026-06-14, branch `core-projection-moneyness-grid`).** `SurfaceConfig`
> gained the validated `moneyness_buckets` field (ordered + symmetric about 0.0 + non-empty),
> authored in `configs/pricing.yaml`. The projection entry points (`run_analytics`,
> `run_analytics_with_qc`, `run_day`, `reconstruct_day`/`reconstruct_range`, the `RunRequest`)
> now default `moneyness_buckets=None` and resolve it from `config.surface` at the single choke

> `pricing` config-hash (and the folded whole-config hash) moved, regenerated by design. Tests:
> `test_projection_moneyness_config.py` (config drives the persisted cells; explicit override;
> default unchanged) + core validator/hash-fold cases. Gate green: ruff/mypy(229)/lint-imports,
> 1934 passed / 12 skipped.

> **Source:** ADR 0028 ("economic inputs live in versioned config, never as scattered
> constants") + TARGET §5.3 (the surface engine's persisted grid). Split out of
> [core-pricing-config-completeness](core-pricing-config-completeness.md): the TASKBOARD line

## The gap (measured)

`DEFAULT_MONEYNESS_BUCKETS = (-0.2, -0.1, 0.0, 0.1, 0.2)` lives as a `.py` literal at
`packages/infra/src/algotrading/infra/actor/driver.py:124`. It is the **log-moneyness grid the
regularized surface is projected and persisted onto** — an economic policy (which strike points
the persisted surface is sampled at, kept symmetric/ATM-centered so the grid is comparable across
underlyings), not a technical constant. It is threaded as a **function default** through the
projection call chain (`driver.py` → `orchestration/reconstruction/batch.py` →
`actor/nautilus_host.py`), so every persisted surface grid is shaped by a literal with **no typed
config home** and **no config-hash coverage** — exactly the ADR-0028 drift this audit class is
about: change the grid and no golden or hash flags it.

Grep confirms: no `moneyness` key in `pricing.yaml` (only `svi_m_bounds`, a different thing); no
field on any `_ConfigModel`; the literal is the only source.

## Scope (core + the read sites)

1. **A typed home.** Add `moneyness_buckets` to `SurfaceConfig` (`pricing.yaml surface:`) — the
   projection is surface output, so `SurfaceConfig` is its natural owner and it already folds into
   `config_hashes["pricing"]`. Validate it is ordered, ATM-centered/symmetric, and non-empty.
2. **Read it where the literal is the default.** The projection call chain reads the configured
   grid instead of the module default. Keep `DEFAULT_MONEYNESS_BUCKETS` only as the in-code
   fallback for callers that pass nothing (tests/notebooks), or retire it if every path threads
   config.
3. **Zero-churn default.** Ship `moneyness_buckets: [-0.2, -0.1, 0.0, 0.1, 0.2]` in `pricing.yaml`
   = byte-identical projection, so no surface golden moves — only the `pricing` config-hash, by
   design (ADR 0028, the C7 pattern the sibling slices used).

## Boundaries

- Core owns the typed field + validation (`platform_config.py`, `pricing.yaml`). The wiring at the
  read sites is **infra** (`infra/actor`, `infra/orchestration`) — this spec spans the core→infra
  seam like `core-explicit-rate-config` did; sequence the infra slice after the field lands.
- Does **not** touch the SVI bounds, the surface model/fallback policy, or the forward engine —
  those are [core-pricing-config-completeness](core-pricing-config-completeness.md)'s remaining
  deferred slices. Coordinate field additions on `SurfaceConfig` to avoid churn.

## Done criteria

The projection moneyness grid is a typed `pricing.yaml` value read by the projection chain (no
`.py` literal at the persisted-grid site); the shipped default reproduces today's grid
byte-identically (only the `pricing` config-hash golden moves, regenerated by design); a unit test
proves a changed config grid changes the persisted surface cells and that the default path is
unchanged; gate green.

## Dependencies

ADR 0028 (accepted — this is an unbuilt slice of it; no new ADR unless placing the field on
`SurfaceConfig` vs a new `projection:` block is judged a contract change at build time).
