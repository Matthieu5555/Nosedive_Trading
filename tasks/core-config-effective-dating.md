# core-config-effective-dating — give config bundles an as-of / effective-dated dimension

> **Source:** TARGET §0 (as-of/effective-dating, a robustness pillar this layer owns) + §6
> (robustness standard: "config is as-of/effective-dated and frozen per run") + ADR 0028
> (configuration & reproducibility standard). The **frozen-per-run** half is built; the
> **as-of/effective-dated** half is not — this task closes that.

## The gap (measured)

`load_platform_config(configs/)` reads the six YAML bundles, builds the typed `PlatformConfig`,
and stamps per-bundle hashes (`config_hashes["pricing" | "qc" | "universe" | …]`) into the run
manifest — so a run is **frozen and reproducible by hash**. What is missing is the **temporal**
dimension ADR 0028 / TARGET §0 promise: a config value has **no `effective_from` / as-of**
notion, so there is no honest answer to *"which `pricing.yaml` was in force on trade_date
2026-06-10?"* other than git archaeology. A replay of an old `as_of` day silently uses **today's**
config, which is a look-ahead leak in the one place the whole platform trusts to be deterministic
(BIG_PICTURE: "provenance and versioning are your guarantee of no look-ahead cheating").

Grep confirms: no `as_of` / `effective_from` field on any `_ConfigModel` in
`core/config/platform_config.py`; `loader.py` has no as-of parameter. The only `effective`
today is `effective_scenario_version` (a version pin, not a date).

## Scope (core only)

1. **An effective-date on the config load path.** `load_platform_config` (and the underlying
   `load_yaml_config` / `from_config`) gain an optional `as_of` so a replay can resolve the
   config **as it stood on that date**, not as it stands now. MVP shape: bundles carry an
   `effective_from` (date) so the loader can select/validate the version in force for an `as_of`;
   absence = "current" (zero-churn default, like `ForwardConfig.rate`'s `null`).
2. **Provenance carries the as-of.** The resolved `as_of` (or "current") joins the per-bundle
   hash in the manifest / `ProvenanceStamp` so a derived record records *which dated config* it
   was computed under — closing the lineage gap, not just the freeze gap.
3. **A guard, not new analytics.** This is a core config/lineage contract: the typed shape, the
   loader resolution, the manifest stamp. It does **not** touch any analytics that consume the
   config (those keep taking a typed `PlatformConfig`, unchanged).

## Boundaries

- Core only: `core/config/*`, `core/provenance.py` / `core/manifest.py`, `configs/*.yaml`
  schema, `packages/core/tests/`. **No** infra/analytics/risk code, **no** rates-curve as-of
  table (that is the *data* as-of, `infra-rates-curve-ingest.md` — a different table, same word).
- Pairs with `core-explicit-rate-config` and `core-pricing-config-completeness` (both add fields
  to the same bundles) — sequence after them or coordinate field additions to avoid churn on
  `platform_config.py`.

## Done criteria

`load_platform_config(configs/, as_of=<date>)` resolves the config in force on that date (current
when unset, byte-identical to today's default path); the resolved as-of is stamped into provenance
alongside the bundle hash; a replay of an old day cannot silently pick up newer config; unit test
proves an `as_of` in the past resolves the dated bundle and a derived record's stamp records it;
`check-lookahead-bias` clean on the loader path; goldens regenerated by design; gate green.

## Dependencies

ADR 0028 (already accepted — this is the unbuilt half of it; no new ADR needed unless the
`effective_from` placement in the YAML schema is judged a domain-contract change at build time).
